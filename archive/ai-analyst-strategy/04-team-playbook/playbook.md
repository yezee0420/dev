# 데이터분석팀 AI 활용 플레이북

## 1. 이 플레이북의 목적

AI를 **개인기가 아닌 팀 표준**으로 만들어, 누구나 동일한 품질의 분석을 더 빠르게 수행할 수 있도록 한다.

## 2. 핵심 원칙

1. **AI 출력은 초안이다** — 반드시 사람이 검증한다
2. **프롬프트는 코드다** — 버전관리하고, 테스트하고, 개선한다
3. **성과는 숫자로 말한다** — 시간, 정확도, 비즈니스 영향으로 측정한다
4. **민감정보는 절대 입력하지 않는다** — 고객식별정보, 계좌번호, 비밀번호 금지

## 3. AI 활용 워크플로우

## 4. 검증 체크리스트

모든 AI 산출물은 아래 항목을 통과해야 최종 배포할 수 있다.

### 수치 검증

- [ ] 소계/합계가 상세 내역의 합과 일치하는가?
- [ ] 전주/전월 대비 증감률 계산이 정확한가?
- [ ] NULL/결측값이 적절히 처리되었는가?
- [ ] 단위(%, 원, 억원, 건)가 올바른가?
- [ ] 소수점 자릿수가 보고 기준에 맞는가?

### 논리 검증

- [ ] 인과관계 주장에 근거 데이터가 있는가?
- [ ] 상관관계를 인과관계로 오해석하지 않았는가?
- [ ] 비교 기준(기간, 세그먼트)이 동일한가?
- [ ] 이상치/극단값이 결과를 왜곡하지 않았는가?

### 보안 검증

- [ ] 고객식별정보(이름, 계좌번호, 주민번호)가 포함되지 않았는가?
- [ ] AI에 입력한 데이터에 민감정보가 없는가?
- [ ] 사내 LLM 정책을 준수했는가?

### 커뮤니케이션 검증

- [ ] 비전문가도 이해할 수 있는 언어인가?
- [ ] 권고안이 실행 가능한 수준으로 구체적인가?
- [ ] 불확실성/한계를 명시했는가?

## 5. 프롬프트 사용 규칙

### 기본 구조

모든 프롬프트는 아래 구조를 따른다:

```
[역할 부여] → [맥락 제공] → [작업 지시] → [출력 형식 지정] → [제약 조건]
```

### 버전관리

- 프롬프트 파일은 `prompt-templates/` 폴더에서 Git으로 관리
- 수정 시 변경 사유를 커밋 메시지에 기록
- 성능이 확인된 프롬프트만 `v{숫자}` 태그를 부여

### 개선 사이클

1. 프롬프트 실행 후 결과 품질을 1~5점으로 평가
2. 3점 이하면 프롬프트 수정 후 재실행
3. 4점 이상이면 팀 공유 프롬프트로 승격
4. 매월 프롬프트 성능 리뷰 (채택률, 수정 빈도)

## 8. FAQ

**Q: AI가 틀린 답을 주면?**
A: 검증 체크리스트로 걸러진다. 통과 못하면 재생성하거나 수동 작성한다. 오류 사례는 반드시 기록한다.

**Q: 어떤 LLM을 써야 하나?**
A: 사내 정책이 허용하는 범위 내에서 선택한다. 정확도가 중요하면 GPT-4o/Claude, 속도가 중요하면 경량 모델.

**Q: 프롬프트를 잘 모르겠다면?**
A: `prompt-templates/` 폴더의 템플릿을 복사해서 데이터만 교체한다. 잘 안 되면 팀 슬랙에 질문한다.

**Q: 자동화했는데 오히려 느려졌다면?**
A: 2주 내 개선이 안 되면 중단한다 (Fail Fast). 원인을 기록하고 다음 과제로 이동한다.
